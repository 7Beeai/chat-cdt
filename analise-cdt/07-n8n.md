## 4. RabbitMQ (roteador inbound)

- **Gatilho:** `rabbitmqTrigger`, fila `cdt`.
- **Agenda:** nenhuma (event-driven).
- **Finalidade:** consome o envelope de webhook da Meta (repassado à fila `cdt`), chama `route_inbound` para decidir o trilho (`cobranca`/`relacionamento`) e **republica na fila correta** (`tatuape` / `tatuape-relacionamento`). Em paralelo registra o inbound para analytics via `record_inbound_message`.

**Tabelas lidas/escritas (no workflow):** nenhuma literal — encapsulado nas RPCs.

| Tabela | Op | Via | Confiança |
|---|---|---|---|
| `message_inbound` | insert | RPC `record_inbound_message` (`p_phone_number_id`, `p_message`) | inferido (no workflow) / **confirmado** na RPC |

**RPCs:**
- `route_inbound` (`p_phone = wa_id`) — lineage real *(de `functions-analysis.json`, `confirmado`)*: lê `clientes_cobranca_setembro` (`unit_id`, `matricula`, `name`, `whatsapp`, `pagamento_feito`, `valor_inadimplente`, `updated_at`), `units` (`rabbitmq_queue`, `id`), `adimplentes_base`; chama `norm_phone_br`; retorna o `rail`/`queue`. Não escreve.
- `record_inbound_message` (`p_phone_number_id`, `p_message`) — lineage real *(confirmado)*: lê `disparadores_whatsapp`; **insere em `message_inbound`** (`wamid`, `from_phone`, `phone_number_id`, `unit_id`, `type`, `body`, `is_optout`, `received_at`, `raw_value`), idempotente por `wamid`.

**Edge functions:** nenhuma. **Serviços externos:** RabbitMQ (consome `cdt`, publica em fila dinâmica `$json.queue`); Supabase REST. Os dois nós de analytics têm `continueOnFail=true`/`neverError=true` (blindados); `Switch Rail` com `fallbackOutput='none'` (mensagem sem rail é descartada).

## 5. Sync Adimplentes - Relacionamento (Motor v2)

- **Gatilho:** `googleDriveTrigger` (`fileCreated`) — **6 pastas** (Grupo Chagas, Vitória, Mogi das Cruzes, Tatuapé, Lavras, Caraguatatuba), uma por unidade.

- **Finalidade:** importação automática de planilhas de **adimplentes** (relacionamento). Detecta arquivo novo, valida a data do nome, baixa o binário e **encaminha (multipart) para um serviço externo** que faz upsert/soft-delete; ao final envia e-mail de resumo.

**Tabelas lidas/escritas / RPCs / edge functions:** **nenhuma** — não há nó Supabase/Postgres nem URL `/rest/v1/` ou `/functions/v1/`.

**Serviços externos:**

- **Serviço HTTP externo customizado** — `POST http://host.docker.internal:8100/adimplentes/sync` (multipart `file`/`file_name`/`file_id`, `httpHeaderAuth 'Sync Adimplentes'`, timeout 600s, retry 2x). **NÃO é edge function Supabase**; toda a persistência (upsert + marcação `bi_atual=false` de quem saiu) acontece **fora do n8n e do Supabase**, neste serviço Docker.

## Integração n8n ↔ CHAT-CDT (DESTAQUE)

Os dois workflows de IA — **CDT Cobrança - Tatuapé-SP (#1)** e **CDT Relacionamento - Tatuapé (#2)** — são o ponto de contato real entre o n8n e o CHAT-CDT. Eles tocam o esquema do CHAT-CDT por **dois mecanismos**:

### a) `chat_record_outbound_message` — espelha o outbound da IA em `messages`

Nó Postgres `registrar_outbound_chat` (credencial **Postgres Supabase**) chama:

```sql
select public.chat_record_outbound_message(
  number_instance, clientId, 'text', <jsonb body>, message_id, 'sent', 'ai'
);
```

A função *(`functions-analysis.json`, writes `confirmado`)*: resolve a unidade pelo `phone_number_id` (`chat_phone_numbers`→`wabas`), faz **upsert de `contacts`** (`unit_id`, `wa_id`, `name`), **garante `conversations` aberta** (insert com tratamento de `unique_violation`→re-SELECT, race contra n8n) e **insere em `messages`** (`conversation_id`, `wa_message_id`, `direction`, `type`, `payload`, `sent_by='ai'`, `status`; `ON CONFLICT (wa_message_id) DO NOTHING`). É o que dá contexto ao operador no handoff.

### b) `agent-tools` — handoff (`transfer_human`) e gate (`ai_may_send`)

Edge function única (1 slug) chamada com `action` distinta *(`edge-functions.json`)*. Auth via header `x-agent-tools-secret`; `verify_jwt=false`; usa `SERVICE_ROLE_KEY` (bypassa RLS):
- `transfer_human` — busca `contacts.id` por `unit_id`+`wa_id` e seta **`conversations.routing='queued'`**, `handoff_reason=reason` na conversa `status='open'`. **É o handoff.**
- `ai_may_send` — **gate** lido antes de a IA responder: retorna `false` se há conversa `open` com `routing in (queued/human)`. Respeita o atendimento humano em curso.
- `block` / `pause` — disparam `agent_block_customer` / `agent_pause_customer` sobre `clientes_cobranca_setembro` (cadência n8n; **não** mexem no CHAT-CDT).

### DIVERGÊNCIA vs `docs/04-n8n-contract.md`

O contrato previa que o n8n faria **3 SQLs inline crus**. O n8n real produz **os mesmos efeitos, mas encapsulados** em uma RPC + uma edge function. **A intenção bate; muda o *onde* o SQL vive — é refactor, não descumprimento.**

| Contrato (`04-n8n-contract.md`) previa (SQL inline) | Implementação real | Efeito |
|---|---|---|
| Ajuste 1: `INSERT INTO messages … sent_by='ai'` direto após o Graph | RPC `chat_record_outbound_message(...,'ai')` (encapsula contact upsert + conversa + insert idempotente) | **equivalente** (a RPC faz mais: também garante contact/conversa) |
| Ajuste 2: `UPDATE conversations SET routing='queued', handoff_reason=…` direto | edge `agent-tools` action `transfer_human` (faz o mesmo UPDATE por dentro) | **equivalente** |
| Ajuste 3: `SELECT exists(… routing='ai') as ai_may_send` (gate) | edge `agent-tools` action `ai_may_send` (retorna `false` só se `routing in queued/human`) | **semântica divergente no caso sem-linha** (ver abaixo) |

**Divergência semântica real no gate** *(cross-cite: `04-n8n-contract.md` Ajuste 3 vs `edge-functions.json` agent-tools)*: o gate do contrato (`exists(... routing='ai')`) retorna **false quando não existe conversa aberta** → IA **bloqueada**. O `agent-tools.ai_may_send` retorna false **só** se há conversa `open` com `routing in (queued/human)` → **true quando não há linha** → IA **pode enviar**. Diverge exatamente no caso que o próprio contrato sinaliza no "Caso A" (IA fala com contato antes de existir conversa no CHAT-CDT): a implementação real **permite** o envio; o SQL literal do contrato **super-bloquearia**. A implementação real é a mais defensável.

**Sync (#6) ↔ contrato:** o **Sync Planilha Power BI v3** chama **`sync_cobranca_v2`**, que escreve `clientes_cobranca_setembro` + `clientes_cobranca_dashboard` + `spreadsheet_sync_log` — todas tabelas do n8n. Não toca o esquema do CHAT-CDT; consistente com "`clientes_cobranca_*` intocada" do contrato.
